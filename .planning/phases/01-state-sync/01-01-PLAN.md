  <name>Task 1: Fix scheduleReconnect stale closure in useGameSocket</name>
  <files>src/app/gameroom/hooks/useGameSocket.ts</files>
  <action>
In `useGameSocket.ts`, rewrite the `scheduleReconnect` useCallback to eliminate the stale closure on `socketState.reconnectAttempts`.

Current problem (lines 77-105): `scheduleReconnect` reads `socketState.reconnectAttempts` directly from the closure. Because `initializeSocket` captures `scheduleReconnect` at the time it was created, the inner socket event handlers (e.g., the `disconnect` handler that calls `scheduleReconnect`) can use a stale version where `reconnectAttempts` is always 0 — this means `MAX_RECONNECT_ATTEMPTS` is never reached, causing infinite reconnection loops.

Fix: Use the functional update form of `setSocketState((prev) => ...)` inside `scheduleReconnect` to read the current attempt count from `prev` rather than from the closure. Also remove `socketState.reconnectAttempts` from the dependency array (it's no longer needed there).

The rewritten function:
```typescript
const scheduleReconnect = useCallback(() => {
  setSocketState((prev) => {
    // Stop trying after max attempts
    if (prev.reconnectAttempts >= MAX_RECONNECT_ATTEMPTS) {
      return {
        ...prev,
        connectionStatus: "error",
        error: "Max reconnection attempts reached",
      };
    }

    // Calculate exponential backoff delay (max 30 seconds)
    const delay = Math.min(
      RECONNECT_DELAY_BASE * Math.pow(2, prev.reconnectAttempts),
      30000
    );

    // Schedule the reconnection attempt
    reconnectTimeoutRef.current = setTimeout(() => {
      initializeSocket();
    }, delay);

    // Update state to show reconnecting status
    return {
      ...prev,
      connectionStatus: "reconnecting",
      reconnectAttempts: prev.reconnectAttempts + 1,
    };
  });
}, [initializeSocket]);
```

Note: This creates a circular dependency (`scheduleReconnect` depends on `initializeSocket`, and `initializeSocket` depends on `scheduleReconnect`). Break this by holding `initializeSocket` in a ref rather than passing it directly as a dependency:

1. Add `const initializeSocketRef = useRef<() => void>(() => {});` near the other refs at the top of the hook.
2. At the bottom of `initializeSocket`, add: `initializeSocketRef.current = initializeSocket;` — or better, after the `initializeSocket` useCallback definition, assign: `useEffect(() => { initializeSocketRef.current = initializeSocket; }, [initializeSocket]);`
3. In `scheduleReconnect`, call `initializeSocketRef.current()` instead of `initializeSocket()`.
4. Change `scheduleReconnect`'s dependency array to `[]` (empty — it reads state via functional update, and calls initializeSocket via ref).

This breaks the circular dependency while keeping both functions stable across renders.

In `useGameEvents.ts`, update the hook to consume `connectionStatus` from `useGameSocket` and set `loading: true` whenever the connection is uncertain (disconnected OR reconnecting).

Current problem (lines 21 and 30-32):
```typescript
const { onEvent, sendEvent, isConnected } = useGameSocket(gameWsUrl, token);
// ...
useEffect(() => {
  updateGameState({ loading: !isConnected });
}, [isConnected, updateGameState]);
```

When the socket transitions from "connected" → "disconnected", `isConnected` goes from `true` to `false`, the effect fires, `loading` becomes `true`. That's correct.

But when `scheduleReconnect` runs and sets `connectionStatus: "reconnecting"`, `isConnected` is ALREADY `false` — no change — so the effect does NOT fire again. During the reconnecting phase, if something else sets `loading: false`, the stale game UI would show.

Fix: Destructure `connectionStatus` from `useGameSocket` and include it in the loading effect:

```typescript
const { onEvent, sendEvent, isConnected, connectionStatus } = useGameSocket(gameWsUrl, token);

// ...

useEffect(() => {
  const isUncertain = !isConnected || connectionStatus === "reconnecting";
  updateGameState({ loading: isUncertain });
}, [isConnected, connectionStatus, updateGameState]);
```

Also update the return value of `useGameEvents` to expose `connectionStatus` so callers (like `page.tsx`) can optionally display a status message. Add `connectionStatus` to the return object:
```typescript
return { sendEvent, connectionStatus };
```

No changes needed to `page.tsx` at this stage — `connectionStatus` is available if needed for a future UI label but the loading gate already handles STATE-02 via the `loadingAtom`.
