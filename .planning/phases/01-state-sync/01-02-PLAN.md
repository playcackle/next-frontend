In `useGameSocket.ts`, modify the `socket.on("connect", ...)` handler inside `initializeSocket` to detect reconnects and emit `request_state_sync`.

The connect handler currently (lines 128-137):
```typescript
socket.on("connect", () => {
  setSocketState({
    isConnected: true,
    connectionStatus: "connected",
    error: null,
    reconnectAttempts: 0,
  });
  clearReconnectTimeout();
});
```

The `setSocketState` call here uses object form, not functional update form — so it doesn't have access to `prev.reconnectAttempts`. Change it to functional form to detect whether this is a reconnect:

```typescript
socket.on("connect", () => {
  setSocketState((prev) => {
    const isReconnect = prev.reconnectAttempts > 0;
    if (isReconnect) {
      // Request current server state on reconnects
      // Backend listens for this event and responds with lobby_state_sync
      socket.emit("request_state_sync");
    }
    return {
      isConnected: true,
      connectionStatus: "connected",
      error: null,
      reconnectAttempts: 0,
    };
  });
  clearReconnectTimeout();
});
```

**Important behavior notes:**
- `socket.emit("request_state_sync")` is called inside `setSocketState`'s updater function — this is safe because the socket is connected at this point (the `connect` event has fired) and emitting is synchronous before the state update is committed.
- If the backend does NOT support `request_state_sync` as a client-emittable event, this emit is silently ignored — it will not cause errors. The backend's existing behavior of pushing `lobby_state_sync` on connect may handle this automatically.
- Do NOT add `"request_state_sync"` to the `gameEvents` array in `initializeSocket` — it is an outbound-only event, not a received event.
- Do NOT call `socket.emit` outside the `setSocketState` updater if there's a concern about React batching; the pattern above is safe because the socket reference is captured in the `initializeSocket` closure and is valid.

**Change B — Wire round_over to request state sync:**

The `round_over` event sets `isRoundBreak: true` and updates scores/accolades, but does NOT provide `slots`. The `AnswerReveal` component reads `slots` from `gameStateAtom` to display answers. If the client missed the `new_round_started` event (e.g., just joined), `slots` will be empty and `AnswerReveal` shows nothing.

Fix: After `round_over` sets `isRoundBreak: true`, emit `request_state_sync` so the server sends a `lobby_state_sync` with `status: "ROUND_BREAK"` and full `slots`. This uses the same `sendEvent` function that's already available in `useGameEvents`.

Update `handleRoundOverRef` to also request sync:
